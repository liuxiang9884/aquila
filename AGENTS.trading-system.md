# AGENTS

本文件定义交易系统项目的代理协作约定。进入项目工作时，默认先读取并遵守这里的规则。

这是一份通用模板，适用于行情、交易、风控、回报处理、策略执行、回测和低延迟基础设施等交易系统项目。使用前应把其中的占位项替换为项目自己的构建命令、测试命令、文档入口和提交流程。

## 默认约定

- 默认使用中文回答，除非用户明确要求其他语言。
- 文档、设计、审查、计划等 markdown 默认使用中文撰写；代码注释使用英文，除非项目已有不同约定。
- 提交信息建议使用英文；PR 标题和正文可按团队约定选择中文或英文。
- 优先遵循仓库现有的构建系统、目录结构、依赖管理和代码风格，不做与当前任务无关的重构。
- 交易系统默认同时关注正确性、确定性、低延迟、可恢复性和可观测性。
- 对性能、时延、吞吐、并发安全、撮合 / 下单 / 撤单 / 回报行为相关结论，必须基于实际测试、benchmark、profile 或运行证据，不凭主观判断宣称完成。

## 新对话启动

每个新对话进入仓库后，先确认当前分支和最近提交：

```bash
git status --short --branch
git log --oneline -8
```

然后按项目实际情况读取：

```text
AGENTS.md
README.md
docs/onboarding.md
docs/architecture.md
docs/testing.md
```

如果这些文档不存在，先读取项目中最接近的 onboarding、README、架构说明、运行手册和测试说明。读取后以项目当前文档和 `git status` / `git log` 为事实源，不假设本地知识是最新的。

## 结束对话流程

当用户要求结束当前对话、交接或生成下一轮 onboarding 时，默认执行：

1. 运行 `git status --short --branch` 和 `git log --oneline -8`。
2. 对照当前实现、配置和最近提交，更新相关文档。
3. 整理当前状态、已完成事项、验证命令、未完成风险和下一步建议。
4. 至少运行 `git diff --check`。
5. 如果本轮触碰交易行为、并发、回报、风控、恢复或性能边界，运行对应测试 / benchmark / smoke。
6. 按团队约定提交文档或代码变更；除非用户明确要求，不主动 push。
7. 最终回复给出提交哈希、验证结果和下一轮接手提示。

如果团队不希望自动提交，应在项目级规则中明确写出。

## 项目背景

交易系统通常包含以下模块：

- 交易所接入与协议适配
- 行情接入、解码、标准化和分发
- 下单、撤单、改单和批量交易接口
- 订单回报、成交回报、账户和仓位处理
- 订单管理、仓位管理、风控、限流和熔断
- 策略执行、信号处理、事件调度和状态恢复
- 回测、replay、benchmark、压测和监控

因此，协作代理工作时应优先关注链路边界、事件顺序、状态一致性、幂等性、异常恢复和验证证据。

## 高频 / 低延迟设计原则

- 默认把主路径低延迟和尾延迟可控作为高优先级目标；吞吐、资源利用率和工程便利性不能破坏主路径确定性。
- 评估方案时优先关注 p50 / p99 / p99.9、抖动、排队和最长停顿，不只看平均值。
- 如果某个方案能提升吞吐量，但会增加排队、批处理、额外拷贝、额外同步、动态分配或尾延迟，应谨慎采用。
- 热路径中避免不必要的动态分配、字符串格式化、日志、虚函数、锁竞争、共享状态写入和大对象拷贝。
- 对初始化后不变、且会在热路径中反复使用的信息，应在启动期解析、校验和缓存。
- 交易行为的正确性、确定性和可恢复性优先级高于单点延迟优化；不能为追求几纳秒牺牲订单状态一致性。
- 性能优化必须先定位瓶颈，再通过 benchmark、profile、链路压测或线上观测验证收益。

## 交易链路设计原则

- 行情接入、策略执行、订单发送、订单回报和风控应有清晰的线程 / 进程边界。
- 策略应表达交易意图，不直接解析交易所 wire protocol。
- 订单管理模块应是订单状态 owner，统一处理本地订单、ack、accepted、rejected、partial fill、filled、cancelled 和 unknown state。
- 下单通道负责发送请求和解析轻量响应；订单回报通道负责处理交易所事实回报；二者不要混淆职责。
- ack 只表示交易所接口收到请求，不等价于订单进入订单簿，不应直接推导持仓变化。
- 持仓变化应由成交回报、账户 / 仓位事件或 reconcile 结果驱动。
- 断线、gap、queue full、reconnect、进程重启后必须明确状态可信度和恢复策略。
- 涉及重试、撤单、replace、cancel all、reduce-only、post-only、IOC/FOK、self-trade prevention 等行为时，应明确交易所语义和本地状态转换。

## 并发和进程模型

- 每个模块应明确线程 owner、对象生命周期、同步方式和内存可见性假设。
- 单 owner thread 可以显著降低共享状态复杂度；跨线程通信优先使用清晰的队列、ring buffer、SHM 或 event channel。
- 热路径避免跨线程阻塞调用；必要时用非阻塞队列和明确的 backpressure / drop / gap 语义。
- 多进程链路必须定义 ABI、版本、容量、ownership、heartbeat、recovery 和 stale owner 处理方式。
- 并发修改必须配套测试或最小复现实验，不能靠代码阅读宣称线程安全。

## 错误处理和恢复

- 区分本地错误、交易所拒绝、网络断线、协议解析失败、队列溢出、状态 gap 和未知订单状态。
- 不要在断线时伪造 filled / cancelled / rejected 等业务事实；应显式标记 unknown 或 degraded，并通过 reconcile 恢复。
- 所有 gap / degraded 状态都应有策略侧行为：暂停新开仓、限制撤单、触发 reconcile、报警或人工介入。
- 恢复流程应明确哪些状态可信、哪些状态需要从交易所重新查询、哪些状态需要人工确认。

## 测试和验证

- 修改交易所协议、行情处理、订单状态机、风控、恢复逻辑或线程模型时，优先补充或更新测试。
- 涉及顺序、幂等、重试、断线重连、超时、账户状态或成交回报处理的变更，必须显式验证结果。
- 新增 parser / encoder 时，至少覆盖正常样例、缺字段、字段类型错误、未知枚举、边界数值和 malformed payload。
- 新增订单状态转换时，至少覆盖 accepted、rejected、partial fill、filled、cancelled、cancel after partial、duplicate / stale feedback 和 unknown order。
- 性能结论必须注明机器、CPU、kernel、编译类型、affinity、benchmark 命令和关键结果。
- live exchange smoke 必须明确是否会真实下单，并给出风控限制、撤单 / 平仓策略和事后账户复核。

## C++ 编码和依赖使用约定

如果项目使用 C++，建议采用以下规则，并按团队现有约定调整：

- 新写或修改的项目自有 C++ 代码，命名遵循团队选定的统一风格；缩进、换行、include 排序等机械格式遵循仓库根目录 `.clang-format`。
- 第三方代码、生成代码和交易所 schema 生成物优先保持上游或生成器输出风格。
- enum 转字符串优先使用项目统一工具或库，避免重复手写 enum-to-string 的 `switch`。
- map / unordered map 语义优先使用项目统一容器；热路径容器选择应基于数据规模、访问模式和 benchmark。
- 打印输出和字符串格式化使用项目统一格式化库；低延迟热路径中避免不必要的临时 `std::string`。
- 需要写入已有缓冲区、避免动态分配或控制截断行为时，优先使用固定缓冲区和明确长度返回。
- 不新增新的格式化 / 打印路径，除非有明确兼容性或系统接口理由，并保持局部化。

## 文档约定

- 设计文档应说明模块职责、数据流、线程 / 进程模型、失败模式、恢复策略、测试计划和性能验证口径。
- 文档中不要把 benchmark micro result 写成端到端交易延迟结论。
- 对外部交易所行为、API 语义、字段含义或限制规则，优先引用官方文档或实际运行证据。
- 当实现边界变化时，同步更新 README、onboarding、架构文档和运行手册。

## Git 和提交约定

- 修改前先看 `git status --short --branch`，确认是否存在用户或他人未提交改动。
- 不要 revert、reset 或覆盖自己没有创建的改动，除非用户明确要求。
- 提交保持原子性：实现、文档、benchmark、清理如果不是同一项最小闭环，优先拆分提交。
- 提交前至少运行 `git diff --check`，并运行与变更相关的最小测试。
- 除非用户明确要求，不主动 push。

## 接手注意事项

- 先确认当前分支、未提交改动和最近提交。
- 先读 onboarding / README / 架构文档，再修改代码。
- 如果需求涉及交易行为、资金、账户或实盘链路，先确认 dry-run / live-run 边界。
- 如果需要真实下单、访问交易所、使用 API key 或启动长期进程，必须先获得用户明确授权。
- 如果发现文档和代码冲突，以当前代码和实际验证为准，并同步修正文档。

## 一句话原则

先确认链路和约束，再实现；先验证根因，再修复；先跑证据，再宣称完成。
