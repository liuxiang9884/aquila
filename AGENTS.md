# AGENTS

本文件定义 `crux` 项目的项目级代理协作约定。进入本仓库工作时，默认先读取并遵守这里的规则。

## 默认约定

- 默认使用中文回答，除非用户明确要求其他语言。
- 优先遵循仓库现有结构、测试方式和数据约束，不做与当前任务无关的重构。
- 对数据结果相关修改，结论应基于实际验证证据，不凭主观判断宣称完成。

## 项目背景

`crux` 是一个偏 Python 数据工程的仓库，核心工作集中在：

- 交易所历史行情下载与导入
- 原始 CSV 到内部 schema 的规范化
- Parquet 月级和日级存储
- `manifest` 状态维护
- `reader` / `writer` / `ingest` 一致性验证
- 回归测试与数据结果校验

因此，这个项目最适合使用能够提高调试、验证、测试和变更控制质量的 skills，而不是偏前端或产品设计类流程。

## 推荐默认使用的 Skills

### `systematic-debugging`

优先用于以下场景：

- 单元测试或回归测试失败
- 导入结果和预期不一致
- 月封存后读取异常
- `manifest` 状态和磁盘目录状态不一致
- 某些日期缺失、重复或排序异常

原因：

- `crux` 的问题通常不是“改一行就好”，而是数据链路某一环节出现偏差。
- 先找根因再修复，比反复试错更适合这个仓库。

### `verification-before-completion`

优先用于以下场景：

- 改完代码后确认是否真的修好
- 准备结束任务前重新跑验证
- 避免“代码能运行但结果不对”

原因：

- 数据工程项目最危险的情况之一，是没有新鲜验证证据就宣称完成。
- 这个 skill 强调依据实际运行结果下结论。

### `test-driven-development`

优先用于以下场景：

- 新增导入逻辑
- 修复 `reader` / `writer` / `ingest` 边界行为
- 增加新数据类型支持
- 修正 `append_active_day()`、`seal_month()`、`get_range()` 等关键逻辑

原因：

- 仓库已经有一批回归测试，适合继续强化先写失败测试、再修实现的方式。
- 对日期边界、月状态切换、幂等写入这类逻辑，TDD 的收益很高。

### `writing-plans`

优先用于以下场景：

- 中等及以上规模改动
- 需要拆分多步任务
- 改动涉及多个模块

典型场景：

- 支持新的交易所
- 增加新的 `data_type`
- 重构 `manifest`
- 增加月重建或恢复流程
- 调整目录结构或 schema 约束

### `brainstorming`

优先用于以下场景：

- 需求还不清楚
- 想增加能力但还没决定方案
- 多种设计都可行，需要先比较取舍

典型场景：

- 增加实时流处理
- 增加更多交易所适配层
- 设计更严格的数据校验机制
- 设计数据修复、重跑、补洞流程

## 次优先级 Skills

### `requesting-code-review`

适合做完较大改动后的结构性复核，用于提前发现边界条件遗漏、测试覆盖不足或实现偏离需求。

### `subagent-driven-development`

适合在已有明确计划且子任务边界清晰时并行推进，不作为这个仓库的日常默认首选。

### `dispatching-parallel-agents`

适合多个互不相关的问题并存，或多个失败点可独立调查时使用。

## 不建议默认启用的 Skills

### `using-superpowers`

不建议作为 `crux` 的默认入口流程。原因是它会把大量任务拉入一套非常严格的流程，对当前以直接协作、快速修正和仓库内实操为主的工作方式来说偏重。

### `using-git-worktrees`

对多人并行开发或长生命周期功能分支很有用，但不是当前项目的第一优先级。

### `finishing-a-development-branch`

更适合正式分支收尾、合并、提 PR 的流程。如果当前主要目标是本地迭代和验证，触发频率通常不会高。

## 推荐工作顺序

通常按下面顺序考虑：

1. 需求没想清楚时，先用 `brainstorming`
2. 需求清楚但改动较大时，用 `writing-plans`
3. 开始实现关键逻辑时，优先 `test-driven-development`
4. 出现异常或失败时，使用 `systematic-debugging`
5. 准备收尾时，必须做 `verification-before-completion`

## 项目级执行规则

- 修改核心数据路径、schema、reader、writer、ingest 或 manifest 行为时，优先补充或更新测试。
- 涉及数据完整性、顺序、去重、边界日期或月封存状态的变更，必须显式验证结果。
- 当函数参数的类型和格式在接口层已经明确时，优先在类型标注、参数注释或 doc 中说明约束，不额外增加参数类型或格式检查逻辑。
- 做性能优化时，先用 benchmark、profile 或明确的热路径分析锁定瓶颈，避免凭直觉增加复杂度。
- 对初始化后不变、且在热路径中会反复使用的信息，优先在构建阶段缓存，不要把重复判断留在每次查询或每次读取中。
- 优先为高频且可明确判定的主路径增加快路径；对低概率场景不要过早做泛化优化，避免为罕见分支引入长期复杂度。
- 优先减少热路径中的额外分配、填充、拷贝和中间结果构造；如果结果区域会被完全覆盖，优先考虑直接分配而不是先填默认值。
- 热路径优化完成后，必须补充行为不变的回归验证，并重新跑 benchmark，确认收益真实存在且没有引入回归。
- 修改应尽量最小化，避免顺手引入无关重构。
- 如果工作区已有未提交更改，先读取并理解现状，再在现有基础上做最小修改。
- 提交应保持原子性：代码实现、文档同步、benchmark 或额外整理如果不是同一项最小闭环，优先分开提交；当一项实现已验证完成时，先单独提交代码，再继续后续文档或 benchmark 修改，避免把中间状态混进同一提交。
- 任何一次修改完成后，默认自动提交到当前 branch；提交前仍应确保本次修改已有对应验证证据，且不要裹带无关改动。

## 一句话原则

先设计，后实现；先验证根因，再修复；先跑证据，再宣称完成。
