## 目标

本文件用于沉淀 `aquila` 的整体架构设计，先从需求讨论开始，逐步收敛到可执行的系统设计。

当前阶段采用问答驱动方式推进：

- 用户未明确提问题时，以记录和整理现有结论为主

## 已知背景

- `aquila` 面向 crypto 高频交易系统
- 当前工程以 `CMake + C++20` 为主构建方式
- 设计默认同时关注正确性、确定性、低延迟、可恢复性和可观测性

## 第一阶段范围

基于当前 [project_structure.md](./project_structure.md) 的约束，第一阶段范围暂定为：

- 接收 `binance` 和 `gate.io` 的永续合约 `BBO` 行情
- 在 `gate.io` 执行永续合约下单交易

## 第一版需求定义

第一版需求先聚焦一个最小可开发闭环，按以下目标推进：

- 行情侧只接入两个交易所的永续合约 `BBO` 行情
- 交易侧只在 `gate.io` 接入永续合约交易能力
- 当前开发、验证和后续架构设计都以这个具体场景为例展开
- 后续如果扩展到更深盘口、更多交易所或更多交易所交易能力，作为后续版本处理

## 当前已知结构边界

- `core`: 交易所无关的系统内核能力

- `test`: 回归验证与测试

#### 记录 3

- `wss://` 表示 `WebSocket over TLS`，是在 `WebSocket` 外层增加 `TLS` 加密通道
- `ws://` 表示明文 `WebSocket`，不提供传输加密，也不提供基于证书的服务端身份校验
- 两者在应用层 `WebSocket` 语义上相同，但握手阶段和底层传输安全属性不同
- 对交易所公网接入场景，默认应优先使用 `wss://`
- 如果把交易所提供的 `wss://` 地址直接改成 `ws://`，即使路径一致，也不代表服务端一定提供明文入口
- 对 `aquila` 第一版设计，交易所公网行情和交易连接默认按 `wss://` 处理

#### 记录 4

- 对 `wss://fx-ws.gateio.ws/v4/ws/usdt` 的实际测试结果是握手成功
- 对 `ws://fx-ws.gateio.ws/v4/ws/usdt` 的实际测试结果是 `ConnectionRefusedError`
- 原因上应优先理解为协议前缀变化同时改变了默认端口：`wss://` 默认连 `443`，`ws://` 默认连 `80`
- 当前测试结果说明目标主机未对该地址暴露可用的明文 `ws://` 入口，至少默认 `80` 端口未提供该服务
- 即使显式改端口后尝试明文 `ws`，交易所公网接口通常也会要求 `TLS`，因此设计上仍应默认按 `wss://` 接入

#### 记录 5

Gate private 接入地址记录：

| 用途 | URL |
|---|---|
| 现货 WebSocket v4 | `wss://spotws-private.gateapi.io/ws/v4/` |
| 衍生品 WebSocket v4 | `wss://fxws-private.gateapi.io/v4/ws/usdt` |
| 衍生品 WebSocket v4 SBE | `wss://fxws-private.gateapi.io/v4/ws/usdt/sbe` |
| API v4 HTTP | `https://apiv4-private.gateapi.io` |

其中三个 `wss://` 地址用于 WebSocket cold path / live probe；`https://apiv4-private.gateapi.io` 是 HTTP REST base endpoint，不进入 WebSocket probe。
