- 记录位置: `doc/`

## 文档定位

本文件定义 `Linux x86_64` 单机部署场景下，面向低延迟、高确定性和可恢复性的通用 `C++ WebSocket client` 最小可接受架构。

本设计是连接内核设计基线，不绑定具体第三方库或交易所协议实现。后续具体交易所适配可在本设计之上承载 `JSON` 和 `SBE` 两类上层 codec。

## 目标

- 给出一个面向低延迟、低抖动、可恢复的 `WebSocket client` 最小可接受架构。
- 约束线程边界、连接 owner、发送路径、接收路径、控制面、内存策略和状态机粒度。
- 为后续实现、benchmark、压测和代码评审提供统一标准。
- 保持 transport-agnostic 的核心设计，允许同一内核承载 `ws`、`wss` 以及不同上层 codec。

## 非目标

- 不在本设计中绑定具体第三方库或框架。
- 不把 `HTTP REST`、管理接口、监控面板纳入同一设计范围。
- 不在本阶段展开交易所私有协议细节、业务字段语义或恢复补数逻辑。
- 不把开发效率优先的通用封装作为目标，默认优先级是低延迟、确定性和尾延迟可控。

## 硬约束

- 单连接始终只有唯一 `I/O owner`，不跨线程执行收发。
- 最小可接受架构允许一个 `I/O thread` 管理多个连接，但关键连接可升级为单连接独占线程并绑核。
- 收发热路径不允许额外线程切换；非 owner 执行流不直接写 socket。
- 同一连接的实际写操作只能由 `I/O owner` 执行；core 内部仅维护严格有界的 `pending write chain` 处理部分写和串行推进。
- 接收路径完成 frame 组装后，应尽可能交付结构化视图或零拷贝字段视图，而不是原始字符串复制。
- `WebSocket core` 不内建业务层 `inbound/outbound queue`；消息如何在线程间移交由集成方决定。
- 心跳、超时判定、退避重连等控制逻辑不应污染热路径，应由独立控制面负责决策。
- 热路径禁止临时分配；buffer、发送节点和解析 scratch 必须预分配或池化。
- `pending write chain` 必须严格有界并采用 `fail-fast`；业务层队列、丢弃或覆盖策略仅作为集成方的低优先级可选扩展。
- 内部状态机必须细粒度，能够区分 `DNS`、`TCP`、`TLS`、`WS`、认证、活跃、退化和重连阶段。
- 可观测性目标按细粒度设计，但生产常开仅限 `L0` 加关键 `L1`，避免观测反向污染延迟。

## 最小可接受连接架构

### 总体结构

- 一个 `I/O thread` 可以管理多个连接。
- 每个连接在任意时刻都只有一个唯一的 `I/O owner`。
- `I/O owner` 负责该连接的全部网络动作：读、写、`TLS` 驱动、`WebSocket frame` 组装、最小 codec 解析、握手推进。
- 业务线程和控制线程都不能直接碰 socket。

### 连接内部分层

- `Transport adapter`
  负责 `TCP` 或 `TLS` 读写抽象，对上提供统一字节流接口。
- `WebSocket session core`
  负责握手、frame 编解码、`ping/pong`、关闭流程、连接状态推进。
- `Codec shim`
  负责最小协议解析，把完整消息转换为结构化视图或零拷贝字段视图。
- `Message delivery contract`
  `I/O owner` 在完成最小解析后，通过 owner-thread callback、`MessageSink` 或轮询接口把消息视图交给集成方；是否入队由集成方决定。
- `Pending write chain`
  连接内部的有界挂起写链，用于串行发送、部分写续写和显式背压。
- `Owner command channel`
  控制面或集成方向 `I/O owner` 注入 `ping`、`close`、`reconnect` 等动作的最小命令通道，不等同于业务消息队列。
- `Control plane`
  独立负责心跳策略、超时判定、退避重连、恢复决策，再把动作投递给 `I/O owner` 执行。

### 数据流形态

```text
user-owned submit path -----> [I/O owner / WS core / transport] -----> network
                                     ^
                                     |
                           control-plane commands
                                     |
network ---------------------------->|
                                     |
                                     v
                           minimal parse / view build
                                     |
                                     v
                  owner-thread callback / sink / poll contract
                                     |
                                     v
                         user-owned delivery path
```

## 线程模型、CPU 绑定与 owner 边界

### 线程模型

- 最小可接受架构采用少量 `I/O threads` 加每连接唯一 owner 的模型。
- 一个 `I/O thread` 可以管理多个连接，但每个连接在生命周期内始终只归属于一个 `I/O thread`。
- 下游业务消费线程与 `I/O thread` 分离。
- 控制面线程或控制模块与 `I/O thread` 分离，不在热路径上执行心跳和重连决策。

### 连接 owner 原则

- 连接建立后，要么固定绑定某个 `I/O thread`，要么在重连时显式迁移到新的 `I/O thread`。
- 不允许连接在正常运行期间被多个线程轮流处理。
- 所有 socket 读写、握手推进、frame 组装、发送状态推进都由 owner 完成。

### CPU 绑定建议

- 默认建议 `I/O threads` 绑核，减少调度漂移和缓存污染。
- 对关键连接，可采用单连接独占 `I/O thread` 加独占核的强化部署方式。
- 控制面线程、消费者线程、后台统计线程应与 `I/O core` 隔离，避免竞争 cache 和调度片。
- 如果部署在 `NUMA` 机器上，连接 owner、内存池和消费线程应尽量保持 `NUMA` 本地性。

### owner 边界

- `WebSocket core` 不强制规定 `inbound/outbound` 业务队列，也不绑定用户的线程移交模型。
- 跨线程发送、消息交付和业务背压由集成方决定，可采用 callback、sink、poll、外部队列或共享内存等方式。
- core 内部只保留连接级 `pending write chain` 和最小 owner 命令注入机制，不把业务消息流模型硬编码进连接内核。
- 所有内部挂起结构都必须有界，容量在初始化阶段确定，不允许热路径隐式扩容。

## 接收路径与发送路径的低延迟约束

### 接收路径目标

接收路径的核心目标是以尽可能小的抖动把网络数据转换成可消费的结构化视图，同时不让业务逻辑、字符串分配和多余拷贝污染 `I/O owner`。

### 接收路径约束

- `I/O owner` 从预分配接收 buffer 读取字节流，不在热路径上扩容。
- `Transport adapter` 只做必要的传输层处理；若启用 `TLS`，其额外开销必须被明确视为 transport 成本，而不是 `WS core` 成本。
- `WebSocket session core` 负责 frame 组装、控制帧处理和关闭流程推进。
- 对完整消息，`Codec shim` 只做最小解析，不承担业务逻辑。
- core 通过 owner-thread callback、`MessageSink` 或轮询接口交付结构化视图、字段偏移、`span` 或等价零拷贝视图；是否排队和如何跨线程移交由集成方决定。

### 接收路径禁止事项

- 不允许在每条消息上构造新的 `std::string` 作为默认交付形态。
- 不允许在 `I/O owner` 里直接执行业务重逻辑；如果使用 owner-thread callback 或 sink，其实现必须保持非阻塞、轻量和可预测。
- 不允许为了通用性在热路径上做多层虚函数分发和不必要的 `type erasure`。
- 不允许在收包路径中做阻塞日志、格式化输出和同步等待。

### 最小解析原则

- 对 `JSON` 场景，优先产出字段视图、token 视图或最小必要字段，而不是完整 `DOM`。
- 对 `SBE` 场景，优先产出指向原始 buffer 的结构化只读视图。
- 如果某类协议的完整解析不可避免地产生额外分配，应将其下沉到消费者线程，而不是放在 `I/O owner`。

### 发送路径目标

发送路径的核心目标是严格串行、行为可预测、无额外线程切换、失败语义清晰。

### 发送路径约束

- 所有实际写操作都必须通过 owner-thread `try_send` 或等价 owner-affine 提交接口推进；跨线程提交由集成方自行完成。
- `I/O owner` 是唯一允许调用底层写接口的执行流。
- 同一连接任一时刻最多只有一条活跃写链。
- 写完成之前，不启动下一条写请求。
- `pending write chain` 必须严格有界，发送编码过程应尽量在预分配 buffer 或对象池中完成，避免临时拼接和重复分配。

### 发送路径禁止事项

- 不允许多个线程直接对同一连接并发写。
- 不允许 `pending write chain` 满时阻塞提交方等待可写。
- 不允许在发送热路径中做隐式重试、长时间自旋或后台同步。
- 不允许在一个通用 `send()` API 背后隐藏复杂锁竞争和内部线程迁移。

### 失败语义

- `pending write chain` 满：立即失败并打点，不阻塞。
- 连接不在可发送状态：立即失败或投递到控制面决策，不静默吞掉。
- 底层写失败：由 owner 更新连接状态，并通知控制面进入后续恢复流程。

## 状态机、心跳、重连与控制面设计

### 状态机原则

- 对外接口只暴露粗粒度状态，例如 `Disconnected`、`Connecting`、`Active`、`Closing`。
- 内部控制面必须维护细粒度状态，用于超时治理、恢复决策、故障归因和阶段耗时统计。
- 状态推进的实际网络动作由 `I/O owner` 执行，但状态决策由控制面主导。

### 内部细粒度状态

- `Disconnected`
- `Resolving`
- `TCPConnecting`
- `TLSHandshaking`
- `WsHandshaking`
- `Authenticating`
- `Active`
- `Degraded`
- `Reconnecting`
- `Closing`
- `Closed`

### 状态机约束

- 不同阶段必须有独立超时，不允许只用一个笼统的连接超时覆盖全部过程。
- `DNS`、`TCP`、`TLS`、`WS handshake`、认证失败必须可区分，否则无法做正确恢复策略和性能归因。
- 状态变更必须单向记录并可观测，不能只靠日志文本推断。

### 心跳设计

- 心跳属于控制面策略，不属于消息热路径。
- 控制面负责决定何时发心跳、多久判超时、连续失败多少次进入退化或重连。
- `I/O owner` 只负责执行具体动作，例如发送 `ping`、处理 `pong`、更新时间戳。
- 心跳 `RTT` 可以作为关键 `L1` 指标，但应限制采样和记录开销。

### 心跳约束

- 心跳定时器不应在每条业务消息路径上穿插检查。
- 心跳实现应与正常写链协调，避免因心跳插队破坏发送顺序语义。
- 如果协议允许专用控制帧，应优先使用更轻量的控制路径；如果必须走应用层心跳，也应保持编码和发送开销最小。

### 退化状态

- `Degraded` 不是已经断开，而是连接仍在，但健康度或时延特征已经异常。
- 进入 `Degraded` 的典型条件可以包括：
  - 心跳 `RTT` 持续升高
  - 连续心跳超时接近阈值
  - `pending write chain` 持续接近高水位
  - owner-thread 消息交付持续滞后或被显式拒绝
- `Degraded` 状态的意义是让控制面可以先降级、限流或准备重连，而不是等到彻底断链才反应。

### 重连设计

- 重连决策在控制面完成，真正的断开、重建连接、重新认证动作由 `I/O owner` 执行。
- 重连必须按失败原因分类处理，而不是一律立即重试。
- 至少应区分：
  - 解析失败
  - `TCP` 建连失败
  - `TLS` 失败
  - `WS` 握手失败
  - 认证失败
  - 活跃态断链
- 重连策略必须支持退避和抖动，避免连接风暴。

### 重连约束

- 不允许在读写热路径里直接触发复杂重连流程。
- 不允许把重连逻辑塞成一个失败就 `while retry` 的同步阻塞过程。
- 对认证失败、配置错误、协议错误，应允许控制面进入暂停重试或人工介入状态，而不是无限重连。
- 对活跃态偶发网络抖动，应支持有限快速重试，但必须受预算和次数约束。

### 控制面职责

- 管理状态机
- 管理分阶段超时
- 管理心跳策略
- 管理退避重连
- 管理恢复动作编排
- 汇总关键健康指标并做阈值判断

### 控制面与 I/O owner 的边界

- 控制面发动作，不直接碰 socket。
- `I/O owner` 执行动作，不自行做复杂策略判断。
- 两者之间通过显式命令和状态回报交互，避免隐式共享状态。

## 内存策略、buffer 组织与零分配约束

### 总体原则

- 热路径默认禁止临时分配。
- 初始化阶段可以分配，冷路径可以分配，但收发、frame 组装、最小解析、发送推进这些路径不应依赖动态扩容。
- 低延迟目标下，内存策略不是优化项，而是架构约束的一部分。

### 需要预分配或池化的对象

- socket 读 buffer
- `TLS record buffer` 或等价传输层 buffer
- `WebSocket frame` 组装 buffer
- `pending write` 节点
- 编码 scratch buffer
- 解析 scratch 或 token buffer
- owner 命令注入所需的最小挂起结构
- 状态事件和观测事件的 ring buffer

### 接收 buffer 组织

- 每个连接应至少持有专属接收 buffer 或专属接收 buffer 视图。
- buffer 容量在初始化阶段确定，按目标消息上界和 burst 情况配置。
- 对超出常规容量的异常大消息，不应默认走热路径扩容；应显式触发异常处理、拒收或退化策略。
- 对 `JSON` 场景，可将原始字节保留在接收 buffer 中，由下游视图直接引用；前提是生命周期边界清晰。
- 对 `SBE` 场景，应优先直接基于接收 buffer 构建只读字段视图。

### 发送 buffer 组织

- 每个连接维护专属发送节点池或等价发送对象池。
- 编码输出优先写入预分配发送 buffer，避免运行时反复拼接 `std::string`。
- 对固定格式消息，例如心跳、登录框架、订阅模板，可在初始化阶段预构建模板，并在发送时只填少量变化字段。
- 发送节点从队列进入写链后，其生命周期必须清晰，避免 owner 与上游共享可变内存。

### 零拷贝与视图原则

- 文档默认追求尽量零拷贝，但不追求形式主义。
- 如果零拷贝会显著增加生命周期管理复杂度或引入更高分支成本，可以允许一次受控拷贝；但这必须是显式设计选择，不是默认结果。
- 下游使用 `span`、`string_view`、偏移表、只读结构化视图等形式优先于新分配对象。
- 所有视图对象必须清楚声明其依赖的底层 buffer 生命周期。

### 解析内存约束

- `JSON` 不默认构建完整 `DOM`。
- `SBE` 不默认拷贝 payload 到新对象。
- 最小解析阶段允许使用固定容量 scratch 区，但不允许每条消息临时申请堆内存。
- 若某类协议字段提取无法避免额外分配，应在下游消费者线程中完成，而不是放在 `I/O owner`。

### 对象池约束

- 对象池必须避免全局锁竞争。
- 连接专属池优先于跨连接共享池，除非已验证共享池不会引入额外抖动。
- 池对象归还路径必须可预测，不允许在热路径上做复杂回收策略。
- 池耗尽时的行为必须明确：失败、降级或触发异常，而不是隐式 fallback 到堆分配。

### 禁止事项

- 不允许默认用 `std::string +=` 在热路径上拼消息。
- 不允许依赖 `std::vector` 自动扩容承载 burst。
- 不允许因为观测、日志或错误路径临时创建大对象并混入主收发路径。
- 不允许先写一个通用版，后面再优化分配，因为这通常会把错误内存模型固化进接口。

## 可观测性、诊断开关与性能预算

### 总体原则

- 可观测性必须服务于低延迟系统，而不是破坏低延迟系统。
- 观测层设计目标应足够细，以支持故障归因、阶段耗时分析和尾延迟定位。
- 但生产默认开启的观测必须受预算约束，避免每消息全量打点把系统本身测坏。

### 分层设计

- `L0`：始终开启的基础观测
- `L1`：生产默认开启的关键阶段观测
- `L2`：诊断或压测模式下开启的细粒度观测

### L0：始终开启

- 连接粗粒度状态变化计数
- 各类错误计数
- 收发字节数
- 收发消息数
- 重连次数
- `pending write chain` 高水位
- 当前连接数、活跃连接数、退化连接数

要求：

- 仅使用低开销计数器或高水位统计。
- 不在热路径做日志格式化。
- 不因计数失败或观测组件拥塞而阻塞业务路径。

### L1：生产默认开启

- `DNS`、`TCP`、`TLS`、`WS`、认证阶段耗时
- 心跳 `RTT`
- owner-thread 消息交付延迟
- `pending write chain` 等待延迟
- 关键状态转移事件
- 退化状态进入或退出原因
- 关键失败原因分类计数

要求：

- 打点只放在明确边界，不做每条消息全链路追踪。
- 时间戳采集必须低开销，优先使用适合绑核场景的高效时钟方案。
- 指标应以线程本地累积、批量汇总为主，避免热路径频繁跨线程写共享结构。

### L2：诊断或压测模式

- 每消息接收时间戳
- 每消息解析耗时
- 每消息排队耗时
- 每消息发送推进耗时
- 更细粒度的状态事件轨迹
- 采样率可配置的详细 tracing

要求：

- 默认关闭。
- 开启时必须明确告知会影响测量对象本身。
- 优先采样，不默认全量记录。
- 适用于 benchmark、回归压测、故障复现场景，不作为常驻生产默认配置。

### 观测预算约束

- 热路径不允许因为观测而引入临时分配。
- 热路径不允许因为观测而进行阻塞 `I/O`。
- 热路径不允许因为观测而做复杂字符串拼装。
- 时间戳采集点必须经过审查，避免为了看得更清楚而在每个细枝末节打点。
- 任何新增观测点都应回答两个问题：
  - 它解决哪个具体排障或性能归因问题
  - 它的常开成本是多少

### 日志与指标边界

- 日志用于异常事件、状态切换和人工诊断，不用于承载热路径逐消息观测。
- 指标用于持续健康监控和长期趋势。
- 详细事件轨迹应走独立诊断通道或采样 ring buffer，而不是同步日志输出。

### 推荐实现方式

- 线程本地计数器加周期性汇总
- 连接状态事件 ring buffer
- 采样式时延追踪
- 高水位、低水位和分位数统计分开设计
- 对长稳运行更关注 `p99`、`p99.9` 和异常窗口，而不是只看均值

## 验证标准、Benchmark、长稳运行与验收门槛

### 总体原则

- 对低延迟 `WebSocket client`，不能只验证能连、能发、能收。
- 最小可接受验收必须同时覆盖功能正确性、恢复行为、性能表现、长稳运行和尾延迟可控性。
- 任何低延迟、低抖动、优化有效的结论，都必须基于新鲜 benchmark 或运行证据，而不是代码阅读推断。

### 一、功能正确性验证

至少覆盖以下场景：

- `DNS -> TCP -> TLS -> WS handshake -> Auth -> Active` 正常推进
- 主动关闭与被动断开
- `ping/pong` 或应用层心跳正常
- 出站消息按发送顺序刷出
- 入站消息能被正确 frame 组装并最小解析
- 队列满时按 `fail-fast` 返回明确失败
- 认证失败、握手失败、断链后的状态推进正确
- 重连后连接状态恢复到预期阶段

验收要求：

- 关键状态转换必须可观察。
- 功能测试不仅看最终成功，还要检查状态机路径是否正确。

### 二、恢复与异常路径验证

至少覆盖以下异常：

- `DNS` 不可解析
- `TCP` 建连失败
- `TLS` 失败
- `WebSocket handshake` 失败
- 认证失败
- 活跃态网络断开
- 心跳超时
- `pending write chain` 持续高水位
- 控制面触发重连退避

验收要求：

- 不同失败原因被归类到不同状态或错误码。
- 重连不会阻塞热路径。
- 配置错误类问题不会进入无限快速重试。
- 网络抖动类问题有受控退避和恢复策略。

### 三、性能 Benchmark

必须至少包含两类 benchmark：

- 微基准
  - frame 编解码耗时
  - 最小解析耗时
  - 出站编码耗时
  - 队列入队和出队耗时
  - 不同 buffer 策略和对象池策略的开销对比
- 链路基准
  - 单连接持续收包延迟
  - 单连接持续发包延迟
  - 单连接双向并发时延
  - 多连接共享一个 `I/O thread` 时的抖动变化
  - `ws` 与 `wss` 的额外开销对比
  - `JSON` 与 `SBE` 两类 codec 的路径开销对比

验收要求：

- 不只看均值，必须给出 `p50`、`p99`、`p99.9`。
- benchmark 必须注明 CPU 绑定、连接数、消息大小、消息频率、是否启用 `TLS`、是否启用 `L2` 观测。
- 任何优化结论必须在同等条件下做前后对比。

### 四、长稳运行验证

必须有长时间稳定性验证，而不是只跑短时压测。

建议覆盖：

- 长时间保持连接活跃
- 周期性心跳
- 持续收发消息
- 间歇性网络抖动或控制面触发重连
- `pending write chain` 高低水位变化
- 内存占用趋势
- 连接对象、buffer、对象池是否存在泄漏或异常增长

验收要求：

- 长稳运行期间内存占用无持续爬升。
- `pending write chain` 水位在合理范围内波动。
- 没有异常状态抖动或错误计数持续增长。
- 长稳状态下尾延迟不出现无解释恶化。

### 五、观测与验证联动

验证必须说明观测层级：

- `L0/L1` 常开下的性能结果
- `L2` 诊断模式下的性能结果
- 若 `L2` 开启导致明显开销，应明确说明，不得与常规生产基线混淆

验收要求：

- 生产基线必须基于 `L0` 加关键 `L1`。
- 诊断模式数据只能用于问题定位和细分分析，不能直接冒充生产性能结果。

### 六、最小可接受验收门槛

一个实现只有在满足以下条件时，才可认为达到最小可接受：

- 功能路径完整，关键状态推进正确
- 异常与恢复路径完整，失败原因可区分
- 热路径没有未经证明的临时分配、阻塞和额外线程切换
- 发送路径严格串行，队列与背压行为符合设计
- benchmark 证明其在目标部署条件下具备可接受的均值和尾延迟表现
- 长稳运行证明其没有明显内存、队列或状态漂移问题
- 可观测性足以支撑故障归因，但不会显著反向污染热路径

## 低优先级可选扩展

- 对语义允许的消息类型引入显式丢弃或覆盖策略
- 集成方按自身线程模型选择外部消息交付或提交流程
- 针对不同 transport 的可插拔优化实现
- 针对不同 codec 的专门解析 fast-path

## 一句话结论

低延迟 `C++ WebSocket client` 的最小可接受架构，应围绕单连接唯一 owner、零额外线程切换、最小解析、严格串行发送、控制面隔离、热路径零临时分配和预算化可观测性展开，并通过功能、恢复、性能和长稳运行四类证据共同验收。
